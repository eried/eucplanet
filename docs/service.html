Service Mode

EUC Planet · diagnostics and protocol probing

Improper use can change wheel settings unexpectedly. Use only when you understand what you are sending.

Service Mode is a hidden screen that records every BLE byte the app exchanges with the wheel and lets you fire test commands by hand. It is meant for users helping to track down a bug, and for protocol research on wheels we don't fully support yet.

Entering and exiting

  1. Tap the version number at the bottom right of the dashboard.
  2. In the About dialog, press and hold the round app logo for 5 seconds.
  3. Confirm the warning. Recording starts and the dashboard's version number pulses red.
  4. To exit, tap the Stop icon in the Service Mode header. The log clears.

While recording, you can close the Service Mode dialog and use the app normally. The log keeps capturing in the background. Tap the red version number to reopen Service Mode at any time.

What is recorded

Each entry has a timestamp and a kind:

  • RX raw bytes received from the wheel (one BLE notification chunk per line)
  • TX raw bytes the app wrote to the wheel
  • CMD a test command fired from the Commands or Raw tab, with its label
  • NOTE internal app events (light toggle requests, reassembled realtime frames, auto-sync events)
  • COMMENT free text the user typed into the comment field
  • INFO session metadata at startup (phone model, Android, RAM, storage, BLE adapter, connected Wear node, wheel firmware)

Nothing is uploaded. The log lives only in memory until you tap Share, which exports it as a .txt via the Android share sheet.

Commands tab

Each connected wheel exposes a list of preset test commands. The label encodes the bytes the app sends, so when you report a result you can say "T6050_FFFF turned the headlight off" and the developer knows the exact packet that worked.

Tap a command, watch the log for the wheel's response, type a comment describing what physically happened, then move on. The dashboard's normal Light, Horn, Lock buttons still work too while Service Mode is open.

Raw tab

Type any sequence of hex bytes and the app sends it. Use this for opcodes the Commands grid doesn't cover yet.

Spaces and dashes are optional. 60 50 01 01, 6050-0101, and 60500101 all parse to the same four bytes. The cap is 64 bytes per send. The Bytes-to-send field below the input is read-only and shows what will actually go on the wire under the current wrap mode.

Wrap mode

The wheel families this app supports use different framing. The wrap mode picker decides what the app does with your typed bytes before sending them.

  • Literal sends the bytes exactly as typed. Use this when you have a complete frame including the aa aa header and XOR checksum already.
  • Wrap V2 treats your bytes as [command] [data...] for the InMotion V2 extended framing. The app prepends aa aa 16 LL 02 21 and appends a XOR checksum. Use this for P6, V12, and V14 control writes when you only know the inner sub-command.
  • Wrap V14 uses the shorter V14 framing with flags 14. Used by some legacy InMotion control writes.

Example. The InMotion P6 light-on command's inner bytes are 60 50 01 01. With Wrap V2 the app expands that to:

aa aa 16 06 02 21 60 50 01 01 [xor]

You only type 60 50 01 01. The framing and checksum are added automatically.

Preset chips

The Insert preset section drops common sub-commands into the input field. Examples for the InMotion V2 family:

  • 60 50 00 00 headlight off (canonical)
  • 60 50 01 01 headlight on
  • 60 51 18 01 horn beep
  • 60 2f 00 / 60 2f 01 auto-headlight off / on
  • 60 24 00 / 60 24 01 25 km/h speed clamp off / on
  • 02 06 info bundle query
  • 02 07 realtime telemetry query
  • 20 20 settings page A query

Combine a preset with the Wrap V2 mode and you can fire any control write the wheel will accept.

Hex pad and undo

The hex pad is collapsible and offers buttons for 0-F plus space, backspace, undo, and redo. Useful when the keyboard layout doesn't have an easy hex input. Undo and redo are multi-step and survive cls.

Adding context to the log

The comment field at the top of Service Mode writes free text into the log between the BLE events. Use it to describe what you observe physically: motor cold = 73F, headlight visibly went off, wheel beeped twice. When the developer reads the shared log, your comments line up with the surrounding RX and TX entries.

Sharing

Tap the Share icon in the Service Mode header. The log goes to a temp .txt file under the app's cache directory and opens the Android share sheet. Send it through any chat app, email, or cloud upload.

For developers extending Service Mode

Each wheel adapter contributes its own preset list via WheelAdapter.getDiagnosticCommands(). To add a new wrap mode, add it to WheelDiagnosticsViewModel.WrapMode, extend the wrap() function with the framing logic, and add a segment in the dialog's wrap-mode row. The list of preset chips is just a hard-coded list at the top of RawTab so it's easy to grow.
